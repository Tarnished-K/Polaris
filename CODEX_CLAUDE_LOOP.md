# Codex–Claude 自律開発ループ運用ガイド

## 1. 目的

Codexを実装担当、Claude CLIを進行管理・レビュー担当として連携させ、次のサイクルを継続するための汎用手順です。

1. Codexがロードマップから作業を実装する
2. Codexが検証結果をClaudeへ送る
3. Claudeが次の作業指示を返す
4. Codexが指示を確認して次の実装へ進む
5. 完成、外部作業待ち、重大な判断待ちのいずれかまで繰り返す

この文書は特定のアプリ、技術構成、ホスティングサービスに依存しません。

## 2. 役割分担

### ユーザー

- 最終的な目的、優先順位、禁止事項を決める
- 外部アカウント、課金、公開範囲など重要な判断を行う
- 必要に応じてループを開始・中断・再開する

### Codex

- リポジトリ、仕様書、ロードマップ、既存コードを確認する
- 実装、テスト、ビルド、レビュー、ドキュメント更新を行う
- 作業完了時にClaude CLIへ簡潔な報告を送る
- Claudeから受け取った指示を安全性と整合性の観点から確認する
- 指示が妥当なら、ユーザーへの中間確認なしで次の作業へ進む

### Claude

- ロードマップと現在の進捗を把握する
- Codexの完了報告を短く評価する
- 次に実施すべき具体的な作業を、必要なら複数まとめて指示する
- 自身でコードを変更せず、進行管理とレビューに集中する
- トークン使用量を抑え、長い説明や既知情報の反復を避ける

## 3. 基本フロー

```text
ユーザーがループ開始を指示
        |
        v
Codexが仕様・ロードマップ・作業ツリーを確認
        |
        v
Codexがまとまった単位で実装
        |
        v
テスト・ビルド・差分確認
        |
        v
Claude CLIへ完了報告
        |
        v
Claudeが次の作業を指示
        |
        +---- 継続可能 ----> Codexが次の実装へ
        |
        +---- 外部作業待ち -> HANDOFFへ記録して別作業へ
        |
        +---- 完成 --------> 最終検証とユーザー報告
```

## 4. Claudeへ最初に渡すルール

最初の呼び出しでは、次のような初期指示を一度だけ渡します。

```text
あなたはこのプロジェクトの進行管理・レビュー担当です。

最初にリポジトリ内のROADMAP.mdを読み、全体像、現在地、未完了項目、
依存関係を把握してください。

以後、Codexから作業完了報告を受け取ったら、次に行う具体的な作業を
指示してください。指示は1件に限定せず、同時に進めるのが適切な
関連作業を複数まとめても構いません。

ルール:
- ロードマップの順序と依存関係を優先する
- 完了済み作業を繰り返さない
- 返答は簡潔にし、トークン消費を抑える
- 確認が必須でない限り、質問で止めない
- 実装ファイルの直接編集はCodexに任せる
- 次の指示、完了条件、必要な検証を明確にする
- 外部アカウント操作や秘密情報が必要な場合は、その境界を明示する
```

## 5. CodexからClaudeへの完了報告

毎回、全文ログや巨大な差分を送らず、判断に必要な情報だけを渡します。

```text
作業完了報告

完了:
- 実装した機能または修正
- 更新した設計・ドキュメント

検証:
- 実行したテストと結果
- ビルド結果
- 必要なら実機・ブラウザ・API確認結果

変更:
- 主要な変更ファイル

未解決:
- 既知の制約
- 外部作業待ち
- 次工程に影響する注意点

ROADMAP.mdと現在の状態を照合し、次に進める具体的な作業を指示してください。
関連する作業は複数まとめて構いません。回答は簡潔にしてください。
```

## 6. Claudeから期待する応答形式

形式を固定すると、Codexが解釈しやすくなります。

```text
NEXT:
1. 実装する内容
2. 同時に進める関連作業

DONE_WHEN:
- 完了条件
- 必須の検証

CAUTION:
- 依存関係、破壊的変更、外部作業などの注意点
```

完了時は次のように返します。

```text
COMPLETE:
- ロードマップ上の完了範囲
- 残っている外部作業
```

## 7. PowerShellによる呼び出し例

以下は、完了報告をClaude CLIへ送り、応答をファイルへ保存する汎用例です。実際のCLIバージョンに合わせて引数を調整してください。

```powershell
param(
    [Parameter(Mandatory = $true)]
    [string]$ReportPath,

    [string]$RoadmapPath = "ROADMAP.md",
    [string]$PrimaryModel = "",
    [string]$FallbackModel = "",
    [string]$OutputPath = ".agent-loop/claude-next.md"
)

$ErrorActionPreference = "Stop"

if (-not (Test-Path -LiteralPath $ReportPath)) {
    throw "Report file not found: $ReportPath"
}

if (-not (Test-Path -LiteralPath $RoadmapPath)) {
    throw "Roadmap file not found: $RoadmapPath"
}

$outputDirectory = Split-Path -Parent $OutputPath
if ($outputDirectory) {
    New-Item -ItemType Directory -Force -Path $outputDirectory | Out-Null
}

$report = Get-Content -LiteralPath $ReportPath -Raw -Encoding utf8

# 送信前に、代表的な秘密情報の形式を伏せる。
$report = $report `
    -replace '(?i)(api[_-]?key|token|secret|password)\s*[:=]\s*\S+', '$1=[REDACTED]' `
    -replace '(?i)bearer\s+[A-Za-z0-9._~+/-]+=*', 'Bearer [REDACTED]'

$prompt = @"
あなたはプロジェクトの進行管理・レビュー担当です。
リポジトリ内の $RoadmapPath を参照し、以下の完了報告を評価してください。
次の作業、完了条件、注意点のみを簡潔に返してください。
関連する作業は複数まとめて構いません。

$report
"@

function Invoke-ClaudeModel {
    param([string]$Model)

    if ([string]::IsNullOrWhiteSpace($Model)) {
        return (& claude -p $prompt 2>&1 | Out-String)
    }

    return (& claude -p $prompt --model $Model 2>&1 | Out-String)
}

try {
    $response = Invoke-ClaudeModel -Model $PrimaryModel
} catch {
    if ([string]::IsNullOrWhiteSpace($FallbackModel)) {
        throw
    }

    $response = Invoke-ClaudeModel -Model $FallbackModel
}

if ([string]::IsNullOrWhiteSpace($response)) {
    if ([string]::IsNullOrWhiteSpace($FallbackModel)) {
        throw "Claude returned an empty response."
    }

    $response = Invoke-ClaudeModel -Model $FallbackModel
}

$response | Set-Content -LiteralPath $OutputPath -Encoding utf8
```

呼び出し例:

```powershell
.\scripts\report-to-claude.ps1 `
    -ReportPath ".agent-loop/completion-report.md" `
    -PrimaryModel "PRIMARY_MODEL_NAME" `
    -FallbackModel "FALLBACK_MODEL_NAME"
```

モデル名は設定ファイルや環境変数で管理し、文書やコミットへ認証情報を含めないようにします。

## 8. CodexがClaudeの指示を受けた後の判断

Claudeの指示は自動的に正しいとは限りません。Codexは実装前に次を確認します。

- 現在のロードマップと矛盾していないか
- すでに完了した作業を重複していないか
- 実際のコードや設定を誤認していないか
- ユーザーが指定した禁止事項に反していないか
- 秘密情報、課金、データ削除、外部公開を勝手に行わないか
- 作業ツリーにあるユーザーの未コミット変更を壊さないか

軽微な誤りなら、Codexが実際の状態に合わせて補正して続行します。目的が変わる、破壊的操作が必要、公開範囲が変わるなど重大な差異がある場合はループを止めてユーザーへ確認します。

## 9. 作業単位

細かすぎる報告はCLI呼び出しとトークン消費を増やします。1回の作業単位には、同じ目的に属する実装と検証をまとめます。

推奨:

- 1つのユーザーフローを完成させる
- UI、状態管理、テストをまとめて扱う
- 同じデータモデルに依存する変更をまとめる
- ドキュメント更新まで含める

分割すべき場合:

- データ移行や認証変更などリスクが高い
- 外部サービスの設定が必要
- 仕様上の選択で結果が大きく変わる
- 大量の既存変更と衝突する

## 10. 検証基準

作業ごとに、該当する範囲を検証します。

```text
静的検査 → 単体テスト → 統合テスト → ビルド → 実動作確認
```

すべてを毎回行う必要はありませんが、リリース候補では一式を実行します。失敗を隠して完了扱いにせず、原因、影響、代替確認を報告へ含めます。

## 11. Git運用

基本方針:

- 作業開始前にブランチと作業ツリーを確認する
- 関係のない変更をステージしない
- コミットには1つの明確な目的を持たせる
- プッシュ前に差分と検証結果を確認する
- 保護ブランチやレビュー規則がある場合は従う

Claudeへの報告はコミット前でも可能ですが、次工程が前の変更へ依存する場合は、検証済みの節目でコミットして状態を固定すると安全です。

## 12. デプロイ方針

プッシュと本番公開は分けて考えます。

- 自動デプロイの対象ブランチを事前に確認する
- プレビューと本番を区別する
- 本番デプロイは複数作業をまとめ、回数を抑える
- ドキュメントだけの変更で本番ビルドを走らせない設定を検討する
- 本番公開後は実際の公開URLや配信ファイルを確認する

Claudeが「デプロイ」と指示しても、ユーザーの許可範囲や既存の運用ルールを超える場合は実行しません。

## 13. セキュリティ

Claudeへ送らない情報:

- APIキー
- OAuthクライアントシークレット
- アクセストークン
- Cookieやセッション情報
- `.env`の内容
- 個人情報や本番データ

送信ログと応答ログは、必要なら `.gitignore` 対象の専用ディレクトリへ保存します。

```gitignore
.agent-loop/
```

## 14. ループの停止条件

次の場合は自動継続を止めます。

### 正常終了

- ロードマップの対象フェーズが完了した
- 必須検証が成功した
- ドキュメントと引き継ぎ情報が更新された

### ユーザー判断待ち

- 仕様が複数案に分かれ、結果が大きく変わる
- 課金、契約、公開、削除など重要操作が必要
- 本番データの変更が必要

### 外部作業待ち

- OAuthやDNSなど管理画面の設定が必要
- 必要な権限・認証・接続がない
- 外部サービスが停止している

### 異常終了

- 同じ失敗を繰り返して進展がない
- Claude CLIが主・予備モデルとも応答しない
- ロードマップとコードの矛盾を安全に解消できない
- 作業ツリーに判別できない変更がある

## 15. 外部作業待ちの扱い

外部作業が必要でも、プロジェクト全体を止める必要がない場合があります。

1. `HANDOFF.md`などに必要な操作、理由、完了確認方法を記録する
2. ロードマップ上で「コード完了・外部設定待ち」と区別する
3. 依存しない次の作業へ進む
4. 外部設定後に必要なE2E確認を残す

## 16. 推奨ドキュメント構成

```text
ROADMAP.md
  フェーズ、優先順位、完了条件、依存関係

HANDOFF.md
  現在地、外部作業、既知の問題、再開手順

README.md
  セットアップ、通常の開発・検証コマンド

AGENTS.md
  エージェント向けのリポジトリ固有ルール
```

ロードマップには各項目の状態を明記します。

```markdown
- [ ] 未着手
- [~] 実装中
- [x] 完了
- [!] 外部作業待ち
```

## 17. 他プロジェクトへの導入チェックリスト

- [ ] `ROADMAP.md`に現在地と完了条件がある
- [ ] テスト、ビルド、静的検査のコマンドが明確
- [ ] Claude CLIが非対話モードで動く
- [ ] 主モデルと予備モデルを設定した
- [ ] 秘密情報を送らない処理がある
- [ ] 応答ログの保存先を`.gitignore`へ追加した
- [ ] CodexとClaudeの役割を分離した
- [ ] 自動継続してよい操作範囲を決めた
- [ ] 本番公開や破壊的操作の停止条件を決めた
- [ ] デプロイ対象ブランチと自動デプロイ設定を確認した
- [ ] 外部作業待ちを記録する場所を決めた
- [ ] ループ全体の正常終了条件を決めた

## 18. ユーザーからの開始指示例

```text
ROADMAP.mdを基準に開発を進めてください。

各作業単位が完了したら、テストとビルド結果を含む短い報告を
Claude CLIへ自動送信してください。Claudeにはロードマップを確認させ、
次に行う具体的な作業を指示させてください。関連作業は複数まとめて
構いません。

Claudeから指示が返ったら、通常の実装範囲では私への中間報告を待たず、
その作業を開始してください。完了後は再びClaudeへ報告し、このループを
完成または停止条件まで続けてください。

Claudeの応答がない場合は予備モデルへ切り替えてください。
本番公開、課金、データ削除、権限変更などの重要操作は自動化の対象外です。
```

## 19. 運用上の注意

- AI同士の合意は、実装の正しさを保証しない
- ロードマップより実際のコードと検証結果を優先する
- 長時間ループでは定期的にGitの状態と未完了項目を再確認する
- Claudeへ送る情報量を固定し、会話履歴の肥大化を防ぐ
- ユーザーが中断を指示したら、CLI連携を即座に止める
- 再開時は前回の応答を盲目的に使わず、最新状態を読み直す

この仕組みの中心は、Codexが実装責任を持ち、Claudeが短いサイクルで次の優先作業を選び、ロードマップと検証結果を共通の判断材料にすることです。
